üîë 1. Hashing Techniques & Algorithms

What is Hashing?

Hashing is a cryptographic function that converts an input (of any size) into a fixed-size string of bytes. The output is called a hash value or digest. Hashing is a one-way function ‚Äì you cannot reverse it to get the original input.

Key Properties of Hash Functions:
  • Deterministic: Same input always produces same output
  • Fast: Quick to compute hash value
  • One-way: Impossible to reverse (find input from hash)
  • Avalanche Effect: Small change in input produces completely different hash
  • Collision-resistant: Hard to find two inputs with same hash
  • Fixed size: Output size independent of input size

Common Hashing Algorithms

Algorithm Hash Size Status Use Case
MD5 128 bits ‚ùå BROKEN Legacy only (collision found)
SHA-1 160 bits ‚ö†Ô∏è DEPRECATED Git, legacy systems (weak)
SHA-256 256 bits ‚úÖ SECURE Bitcoin, passwords, general use
SHA-512 512 bits ‚úÖ SECURE High-security applications
SHA-3 256/512 ‚úÖ MODERN Future applications, highly secure
BLAKE2 256/512 ‚úÖ FAST Faster than MD5, very secure

Hash Function Example: SHA-256

Input: "Hello World"
Output: a591a6d40bf420404a011733cfb7b190d62c65bf0bcda32b57b277d9ad1f146e

Input: "Hello World!"
Output: 7f83b1657ff1fc53b92dc18148a1d65dfc2d4b1fa3d677284addd200126d9069

Note: Tiny change (added "!") = completely different hash (Avalanche Effect)

Hashing Applications

Integrity Verification

Verify data hasn't been modified. Download files with hash to check integrity.

Password Storage

Store hash of password, not password itself. Never reverse to get password.

Blockchain

Bitcoin uses SHA-256 to create immutable blocks linked by hashes.

Digital Signatures

Hash message, then encrypt hash with private key for signature.

Salting & Hashing for Passwords

Why Salt? Without salt, two users with same password have same hash (vulnerability). Salt adds random value to each password before hashing.
Example:
Password: "password123"
Salt: "abc123xyz"
Hash: SHA-256("password123" + "abc123xyz") = unique hash

Hashing Best Practices

Recommendations:

  • Use SHA-256 or SHA-512 (never MD5 or SHA-1)
  • For passwords, use bcrypt, Argon2, or scrypt (slower, more secure)
  • Always use salt for password hashing
  • Use multiple iterations (e.g., bcrypt with cost factor 12+)
  • Never log passwords or hashes in plain text
  • Implement rate limiting on password attempts

‚úçÔ∏è 2. Digital Signatures

What is a Digital Signature?

A digital signature is a cryptographic technique that proves:

  1. Authentication: Message is from claimed sender
  2. Non-repudiation: Sender cannot deny sending message
  3. Integrity: Message hasn't been modified

How Digital Signatures Work

1

Hash Message: Hash the message using SHA-256 (or similar)

2

Encrypt Hash: Encrypt hash with sender's private key (creates signature)

3

Send: Send message + signature to recipient

4

Decrypt Signature: Recipient decrypts signature with sender's public key (gets original hash)

5

Hash Message: Recipient hashes received message with same algorithm

6

Compare: If hashes match, signature is valid (message authentic & unmodified)

Digital Signature Algorithms

Algorithm Based On Key Size Use Case
RSA Integer Factorization 2048-4096 bits Email, documents, SSL/TLS
DSA Discrete Logarithm 1024-3072 bits Legacy systems
ECDSA Elliptic Curve 256-521 bits Bitcoin, Ethereum (more efficient)
EdDSA Elliptic Curve 256 bits Modern cryptography (fastest)

Digital Signature Example: RSA

Signing Process:
1. Message: "Pay Alice $100"
2. Hash: SHA-256(message) = abc123xyz...
3. Signature: RSA_Encrypt(hash, private_key) = xyz789abc...
4. Send: message + signature

Verification Process:
1. Receive: message + signature
2. Hash_received: SHA-256(message) = abc123xyz...
3. Decrypted: RSA_Decrypt(signature, public_key) = abc123xyz...
4. If Hash_received == Decrypted: Valid signature ‚úÖ

Applications of Digital Signatures

Email Security

S/MIME and PGP use digital signatures to authenticate email.

Document Signing

Legal documents, contracts signed digitally with non-repudiation.

SSL/TLS Certificates

Websites signed with digital certificates (public key infrastructure).

Blockchain

Transactions signed with ECDSA (Bitcoin, Ethereum).

üîë 4. Password Authentication

Password Basics

Passwords are the most common form of authentication. Users provide a secret string that only they should know.

Password Security Challenges

‚ùå Weaknesses of Passwords

  • User Weakness: Users choose weak passwords (123456, password, qwerty)
  • Reuse: Users reuse password across multiple sites
  • Forgetting: Users forget passwords, need recovery
  • Phishing: Users tricked into revealing password
  • Keylogging: Malware captures password as typed
  • Database Breach: If database breached, passwords compromised
  • Shoulder Surfing: Someone watches password being typed

Password Security Best Practices

For Users:

  • Use strong passwords (12+ characters, mixed case, numbers, symbols)
  • Use unique password for each important site
  • Use password manager (e.g., Bitwarden, LastPass) to store passwords securely
  • Never share password with anyone
  • Never use personal information (birthday, name) in password
  • Change password regularly (every 90 days recommended)
  • Avoid common patterns (qwerty, 123456, abc123)

For Administrators:

  • Hash passwords with salt (never store plaintext)
  • Use bcrypt, Argon2, or scrypt (slow, resistant to attacks)
  • Implement rate limiting (limit login attempts)
  • Require strong password policy
  • Implement account lockout after failed attempts
  • Use HTTPS for login (encrypt in transit)
  • Implement password expiration (optional, debated)
  • Enable multi-factor authentication

üëÜ 5. Biometric Authentication

What is Biometric Authentication?

Biometric authentication uses unique physical or behavioral characteristics to verify identity. It's based on "something you are."

Types of Biometric Authentication

Fingerprint Recognition

Most common. Scans fingerprint pattern. Used in phones, buildings, government.

Facial Recognition

Analyzes facial features (distance between eyes, nose shape, etc). Used in airports, phones.

Iris/Retina Scanning

Scans unique patterns in eye. Very accurate. Used in high-security areas.

Voice Recognition

Analyzes voice patterns. Used in banking, phone authentication.

Behavioral Biometrics

Typing pattern, gait (walking pattern), mouse movement. Continuous authentication.

Hand Geometry

Scans hand shape and size. Used in some access control systems.

Biometric Comparison

Biometric Type Accuracy Cost Acceptance Spoofing Risk
Fingerprint 99.9% Low High Medium (fake finger)
Facial Recognition 99%+ Medium High High (deep fake, photo)
Iris/Retina 99.9%+ High Low Very Low
Voice 97% Low Medium Medium (voice recording)
Behavioral 95-98% Low Medium Low

Biometric Advantages & Disadvantages

‚úÖ Advantages

  • No password to forget
  • Very hard to spoof/fake
  • Fast authentication
  • User-friendly (no typing needed)
  • Unique to individual (can't be shared)
  • Difficult to steal or duplicate

‚ö†Ô∏è Disadvantages

  • Expensive equipment
  • Privacy concerns (tracking)
  • Database breach = permanent loss
  • False rejections (10-20%)
  • Cannot change like password
  • May not work for disabled people
  • Spoofing possible (deep fakes, fake fingers)

Biometric Security Concerns

‚ö†Ô∏è Threats to Biometric Systems

  • Spoofing: Fake fingerprint, photo, or voice recording
  • Template Theft: Stolen biometric templates from database
  • Liveness Detection Bypass: Defeating liveness checks
  • Deep Fakes: Synthetic facial/voice generation
  • Presentation Attacks: Presenting fake biometric (silicone finger)
  • Privacy: Tracking people without consent
  • False Positive: System accepts wrong person

üîê 6. Multi-Factor Authentication (MFA)

What is Multi-Factor Authentication?

MFA requires two or more authentication factors to verify identity. Even if one factor is compromised, account remains secure.

MFA Types & Methods

Two-Factor Authentication (2FA)

Requires exactly 2 factors. Example: Password + OTP

Three-Factor Authentication (3FA)

Requires 3 factors. Example: Password + OTP + Biometric

Common MFA Methods

Method Type Factor Security Level Adoption
SMS OTP One-Time Password Something You Have ‚ö†Ô∏è Medium Highest
Authenticator App Time-based OTP (TOTP) Something You Have ‚úÖ High High
Hardware Token FIDO2/U2F Something You Have ‚úÖ Very High Medium
Biometric + Password Hybrid Multiple Factors ‚úÖ‚úÖ Excellent Growing
Push Notification App-based approval Something You Have ‚úÖ High Medium
Email Code One-Time Code Something You Have ‚ö†Ô∏è Medium High

How TOTP (Authenticator Apps) Work

1

Setup: User scans QR code in app (Google Authenticator, Microsoft Authenticator, Authy)

2

Shared Secret: App stores shared secret key synchronized with server

3

Generate Code: App generates new 6-digit code every 30 seconds using algorithm

4

User Enters Code: User enters 6-digit code from app during login

5

Server Verifies: Server generates same code with same algorithm, compares with user input

6

Grant Access: If codes match, access granted (code only valid for 30 seconds)

MFA Best Practices

Recommendations:

  • Use authenticator apps over SMS (SMS susceptible to interception)
  • Use hardware security keys (U2F/FIDO2) for high-security accounts (email, banking)
  • Combine different factor types (not just 2 passwords)
  • Keep backup recovery codes safe (if you lose phone)
  • Enable MFA on all important accounts (email, banking, social media)
  • Don't use SMS for financial institutions (use hardware tokens)
  • Avoid email-only 2FA if possible (use app or hardware token)

MFA Vulnerabilities

‚ö†Ô∏è MFA Can Still Be Bypassed

  • SIM Swap: Attacker convinces carrier to transfer phone number
  • Phishing: User enters password + OTP on fake website
  • Malware: Keylogger captures code as typed
  • Social Engineering: Attacker tricks user into revealing OTP
  • Hardware Token Loss: User loses USB security key
  • Backup Code Theft: Attacker finds written backup codes

üõ°Ô∏è 7. CIA Security Model

What is the CIA Model?

The CIA model (Confidentiality, Integrity, Availability) is the foundation of information security. All security measures should protect these three principles.

The Three Pillars of CIA

üîí Confidentiality

Definition: Only authorized people can access information.

Ensures:

  • Data is not disclosed to unauthorized parties
  • Privacy of sensitive information
  • Secrets remain secret

Threats: Eavesdropping, data breach, unauthorized access

Safeguards: Encryption, access control, authentication

üìù Integrity

Definition: Data is accurate and hasn't been modified.

Ensures:

  • Data is reliable and trustworthy
  • Information hasn't been tampered with
  • Completeness of data

Threats: Data corruption, unauthorized modification, malware

Safeguards: Hashing, digital signatures, checksums

‚ö° Availability

Definition: Information is accessible when needed.

Ensures:

  • Systems are operational
  • Data can be accessed quickly
  • No downtime or service disruption

Threats: DDoS attacks, system failures, disasters

Safeguards: Redundancy, backups, load balancing

CIA Model Examples by Domain

Domain Confidentiality Integrity Availability
Healthcare Patient privacy (HIPAA) Accurate medical records 24/7 system availability
Banking Account details secret Transaction accuracy ATM/banking hours available
E-commerce Credit card encrypted Order details not modified Website always online
Military Top secret classified data Mission accuracy Redundant communications

Implementing CIA in Practice

Security Controls:

  • Confidentiality: Encryption (AES), TLS/SSL, VPN, firewalls, access control lists (ACL)
  • Integrity: Hashing, digital signatures, HMAC, checksums, version control
  • Availability: Load balancing, backup systems, disaster recovery, DDoS mitigation, redundancy

Trade-offs in CIA

Common Conflicts:
  • Confidentiality vs Availability: Heavy encryption slows system (less available)
  • Confidentiality vs Usability: Strong authentication inconvenient (bad UX)
  • Integrity vs Performance: Extra hash checks slower (reduced availability)
  • Solution: Balance security with business needs (risk-based approach)

‚ö†Ô∏è 8. Authentication Threats & Attacks

Common Authentication Attacks

1. Brute Force Attack

Description:

Attacker tries all possible passwords until finding correct one.

Example:

Trying all 4-digit PINs (0000-9999) = 10,000 attempts

Defense:

  • Rate limiting (max 5 attempts per minute)
  • Account lockout (lock after 10 failed attempts)
  • Strong passwords (128-bit entropy)
  • CAPTCHA verification
  • Progressive delays between attempts

2. Dictionary Attack

Description:

Attacker uses list of common words/passwords instead of all combinations.

Why Effective:

Most passwords are weak (password, 123456, qwerty, admin)

Defense:

  • Enforce strong password policy
  • Forbid common passwords (check against dictionary lists)
  • Rate limiting and account lockout
  • Multi-factor authentication

3. Rainbow Table Attack

Description:

Attacker uses pre-computed hash table to instantly look up password from hash.

Example:

Database breached: password hashes stolen. Attacker looks up hash in rainbow table.

Defense:

  • Use salt (makes rainbow tables useless)
  • Use slow hash functions (bcrypt, Argon2)
  • Use unique salt per password
  • Never use MD5 for passwords

4. Phishing Attack

Description:

Attacker tricks user into entering credentials on fake website.

Example:

Email "from bank" with link to fake login page. User enters username/password. Attacker steals credentials.

Defense:

  • Check URL carefully (legitimate: bank.com, fake: bank-secure.xyz)
  • Enable MFA (even if password stolen, can't login)
  • Use password manager (auto-fills only for correct domain)
  • Look for SSL certificate (lock icon in browser)
  • Never click email links (navigate directly to website)
  • Email authentication (SPF, DKIM, DMARC)

5. Man-in-the-Middle (MITM) Attack

Description:

Attacker intercepts communication between client and server.

Example:

User connects to unsecured WiFi. Attacker intercepts login request (password sent in plaintext).

Defense:

  • Use HTTPS/TLS encryption
  • Verify SSL certificate
  • Use VPN on public WiFi
  • Use HSTS (force HTTPS)
  • Certificate pinning (app)

6. Keylogging & Malware

Description:

Malware records every keystroke on user's computer.

Example:

User installs infected app. Malware logs password as typed. Attacker gets password.

Defense:

  • Use antivirus/anti-malware software
  • Keep OS updated (patch vulnerabilities)
  • Use MFA (password alone not enough)
  • Use on-screen keyboard (prevents keylogger)
  • Use password manager (doesn't type password)
  • Use hardware security keys (resistant to keylogging)

7. Credential Stuffing

Description:

Attacker uses stolen credentials from one breach to try on other sites (users reuse passwords).

Example:

LinkedIn breach: 600M password hashes stolen. Attacker tries credentials on Facebook, Gmail, Twitter.

Defense:

  • Use unique passwords for each site
  • Use password manager
  • Enable MFA (prevent unauthorized access)
  • Monitor for breaches (Have I Been Pwned?)
  • Change password if site breached

8. Social Engineering

Description:

Attacker manipulates person into revealing secrets (non-technical hack).

Example:

Attacker calls IT support pretending to be employee. "I forgot password, need reset." IT resets without verification.

Defense:

  • Security awareness training
  • Verify identity before revealing information
  • Never trust caller ID (can be spoofed)
  • Use code words for verification
  • Culture of security (report suspicious behavior)

üì° 9. RPC (Remote Procedure Call) Protocol

What is RPC?

RPC (Remote Procedure Call) is a protocol that allows a program to execute a function/procedure on a remote computer as if it were local. Fundamental to distributed computing.

RPC Basics

How RPC Works:
  1. Client calls a function (appears local)
  2. RPC client library packages function name & parameters
  3. Packages sent over network to remote server
  4. RPC server library unpacks parameters
  5. Server executes function on its machine
  6. Result returned to client over network
  7. Client library unpacks result, returns to caller

RPC vs Regular Function Call

Aspect Local Function Call RPC (Remote Call)
Location Same process/memory Different machine/process
Speed Very fast (microseconds) Slow (milliseconds)
Failure Modes Stack overflow, crash Network failure, timeout
Security Within same process Network exposed (needs encryption)
Parameters Direct references Serialized/marshalled

RPC Implementation Models

Synchronous RPC

Client waits for server response. Function blocks until result received.

Pros: Easy programming

Cons: Client blocked, slow

Asynchronous RPC

Client doesn't wait for response. Callback function invoked when result ready.

Pros: Non-blocking, faster

Cons: Complex programming

RPC Call Semantics

Semantics Description Guarantee
At-Most-Once Call executed 0 or 1 time No duplicates, may be lost
At-Least-Once Call executed 1 or more times Never lost, may be duplicated
Exactly-Once Call executed exactly 1 time Perfect (hard to implement)

Popular RPC Frameworks

JSON-RPC

Simple RPC using JSON over HTTP. Used in blockchain (Bitcoin, Ethereum nodes).

gRPC

Modern RPC by Google. Uses Protocol Buffers + HTTP/2. Very fast.

XMLRPC

Older standard using XML. Less popular now (JSON preferred).

SOAP

Enterprise RPC using XML. Complex but secure. Used in banking/finance.

RPC Example: Blockchain (Bitcoin/Ethereum)

Bitcoin JSON-RPC Call:

Request:
{
  "jsonrpc": "2.0",
  "method": "getbalance",
  "params": ["myaccount"],
  "id": 1
}

Response:
{
  "jsonrpc": "2.0",
  "result": 0.56,
  "id": 1
}

Client calls getbalance() on remote Bitcoin node. Node returns account balance.

RPC Security Considerations

‚ö†Ô∏è RPC Security Threats

  • No Authentication: If not secured, anyone can call RPC functions
  • Network Exposure: RPC exposed to network (firewall needed)
  • Privilege Escalation: RPC calls may execute with server privileges
  • Injection Attacks: Malicious parameters in RPC call
  • Man-in-the-Middle: RPC traffic not encrypted (use TLS)
  • DDoS Attacks: Flood server with RPC calls

RPC Security Best Practices

Securing RPC:

  • Use TLS/SSL: Encrypt RPC traffic over HTTPS
  • Authentication: Require API keys or credentials
  • Authorization: Check user permissions before executing function
  • Rate Limiting: Limit RPC calls per user/IP
  • Input Validation: Validate all RPC parameters
  • Firewall Rules: Only allow RPC from trusted IPs
  • Logging: Log all RPC calls for audit trail
  • Timeouts: Prevent long-running RPC calls from blocking

‚úÖ Summary & Key Takeaways

Key Concepts

Cryptography Fundamentals:

  • Hashing: One-way function converting input to fixed-size hash. Used for integrity & password storage.
  • Digital Signatures: Prove message authenticity & integrity. Sign with private key, verify with public key.
  • Encryption: Protect confidentiality. Symmetric (same key) for speed, asymmetric (public/private key) for distribution.

Authentication Methods:

  • Passwords: Most common but weakest. Susceptible to phishing, brute force, reuse.
  • Biometrics: "Something you are." Fast, hard to spoof, but privacy concerns. Backup authentication option.
  • MFA: Multiple factors (password + OTP + biometric). Essential for high-security accounts.

CIA Model:

  • Confidentiality: Data privacy (encryption, access control)
  • Integrity: Data accuracy (hashing, digital signatures)
  • Availability: System uptime (redundancy, backups, DDoS protection)

Security Recommendations

Work System MFA + single sign-on (SSO) + biometric option

The Future of Authentication

  • Passwordless Future: FIDO2/WebAuthn replacing passwords (already happening)
  • Biometric Integration: Face ID, fingerprint becoming standard
  • Decentralized Identity: Self-sovereign identity (blockchain-based)
  • Continuous Authentication: Behavioral biometrics monitoring (typing pattern, mouse movement)
  • Risk-Based Authentication: Adaptive authentication based on context
  • Quantum-Resistant Crypto: Post-quantum cryptography (preparing for quantum computers)
