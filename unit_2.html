1. Basic Cryptocurrency

Definition

Cryptocurrency is a digital or virtual form of money that uses cryptography for security. It operates independently of a central bank and uses decentralized networks (typically blockchain) to verify and record transactions.

Key Characteristics

üîê Cryptographic Security

Uses advanced encryption algorithms to secure transactions and verify ownership.

‚ö° Fast Transfers

2. Creation of Coins

Coin Generation Methods

Cryptocurrencies are created through various mechanisms, each with different economic models and properties.

1. Mining (Proof of Work)

Process

  • Miners compete to solve complex mathematical puzzles
  • First to solve receives newly created coins (block reward)
  • Transaction fees also go to miner
  • Difficulty adjusts to maintain regular block creation

Example: Bitcoin mining creates 6.25 BTC per block (halving every 210,000 blocks)

2. Staking (Proof of Stake)

Process

  • Validators lock up coins as collateral
  • Selected to create new blocks (weighted by stake)
  • Earn rewards for validating transactions
  • Slashing penalty for misbehavior

Example: Ethereum 2.0 uses Proof of Stake with 32 ETH minimum stake

3. Initial Coin Offering (ICO)

Process

  • Tokens created and sold to raise funds
  • Often used for new blockchain projects
  • Buyers get tokens for development funds
  • Can represent equity, utility, or governance rights

4. Airdrop

Free distribution of tokens to holders of another cryptocurrency or to community members.

3. Payments and Double Spending

Cryptocurrency Payments

A payment in cryptocurrency involves transferring value from one party to another using cryptographic signatures.

Payment Flow

Transaction Structure

Field Description Example
Input (UTXO) Previous transaction outputs being spent 1.5 BTC from transaction X
Output Recipients and amounts 1.0 BTC to Bob, 0.4 BTC change
Signature Proof of ownership (sender's private key) ECDSA signature
Fee Incentive for miners to include transaction 0.1 BTC (input - output)

Double Spending Problem

What is Double Spending?

The problem where digital currency could theoretically be spent twice by creating two transactions using the same funds.

Example of Double Spending Attack

Scenario:

  • Alice has 1 BTC
  • She sends 1 BTC to Bob (transaction 1)
  • She also sends the same 1 BTC to Charlie (transaction 2)
  • Both transactions are broadcast before confirmation
  • Which transaction is valid?

Solution: Blockchain Consensus

How Blockchain Prevents Double Spending

  • Sequential Recording: Transactions recorded in order on blockchain
  • Consensus Mechanism: Network agrees on transaction order (PoW/PoS)
  • Immutability: Once confirmed, transactions cannot be reversed
  • Hash Linking: Each block contains hash of previous block
  • Majority Rule: Attacker needs 51% hash power to rewrite history

Transaction Confirmation

Status Confirmations Security Level Time (Bitcoin)
Unconfirmed 0 Low 0-10 minutes
One Confirmation 1 Medium ~10 minutes
Six Confirmations 6 High ~60 minutes
High Security 100+ Very High 16+ hours

51% Attack

Definition: If an attacker controls 51% of network hash power, they could potentially:

  • Reverse recent transactions (double spending)
  • Prevent other transactions from confirming
  • Monopolize block creation

Cost: Prohibitively expensive for established networks (billions for Bitcoin)

4. FORTH: The Precursor for Bitcoin Scripting

What is FORTH?

FORTH is a stack-based programming language created in 1970. It's minimal, efficient, and influenced the design of Bitcoin Script, making it highly relevant to cryptocurrency development.

FORTH Characteristics

  • Stack-Based: Uses stack for operations (postfix notation)
  • Minimal Syntax: Simple and extensible
  • Efficient: Fast execution and small memory footprint
  • Embedded Systems: Commonly used in embedded systems
  • Concatenative: Functions composed by concatenation

Basic FORTH Example

: SQUARE ( n -- n*n )
    DUP * ;

5 SQUARE ( Result: 25 )
10 3 + . ( Result: 13 )

Stack Operations in FORTH

Operation FORTH Code Stack Effect Description
Push number 5 ( -- 5 ) Pushes 5 onto stack
Addition + ( a b -- a+b ) Pops 2, pushes sum
Duplicate DUP ( a -- a a ) Duplicates top item
Drop DROP ( a -- ) Removes top item
Swap SWAP ( a b -- b a ) Swaps top two items
Multiply * ( a b -- a*b ) Pops 2, pushes product

Why FORTH for Blockchain?

  • Simple and secure (less room for bugs)
  • Stack-based nature suits blockchain validation
  • Minimal bytecode reduces transaction size
  • Efficient execution in nodes
  • Easy to audit and analyze

FORTH vs Bitcoin Script Comparison

5. Bitcoin Scripts

Definition

Bitcoin Script is a simple, stack-based, Turing-incomplete scripting language used to define spending conditions for bitcoins. It determines who can spend unspent outputs.

Key Properties

  • Stack-Based: Uses two stacks (main and alternative)
  • Not Turing-Complete (by design):
    • No loops (prevents infinite programs)
    • Script size limit (10,000 bytes)
    • Operation limit (201 operations max)
  • Deterministic: Always produces same result
  • Public: Spending rules visible on blockchain

Script Types

1. Pay-to-Public-Key (P2PK)

Pays directly to a public key. Recipient proves ownership with private key signature.

<pubkey> OP_CHECKSIG

2. Pay-to-Public-Key-Hash (P2PKH)

Pays to hash of public key (more common). Receiver reveals public key and signs.

OP_DUP OP_HASH160 <hash> OP_EQUALVERIFY OP_CHECKSIG

3. Pay-to-Script-Hash (P2SH)

Allows complex scripts without bloating blockchain. Redeem script provided by spender.

OP_HASH160 <scripthash> OP_EQUAL

4. Pay-to-Witness-Public-Key-Hash (P2WPKH)

SegWit version of P2PKH. Signature data moved to witness field (reduces transaction size).

5. Multi-Signature (M-of-N)

Requires M signatures out of N public keys to spend.

<M> <pubkey1> ... <pubkeyN> <N> OP_CHECKMULTISIG

Common Bitcoin Script Operations

Operation Code Effect
OP_DUP 0x76 Duplicates top stack item
OP_HASH160 0xa9 HASH160 of top item
OP_EQUAL 0x87 Compares top two items
OP_EQUALVERIFY 0x88 Equal, fails if not equal
OP_CHECKSIG 0xac Verifies signature
OP_CHECKMULTISIG 0xae Verifies multiple signatures

Script Execution Example (P2PKH)

Spending a P2PKH Output

ScriptSig (input script): <signature> <pubkey>

ScriptPubKey (output script): OP_DUP OP_HASH160 <hash> OP_EQUALVERIFY OP_CHECKSIG

Execution:

  1. ScriptSig executed: signature and pubkey on stack
  2. ScriptPubKey executed: verifies pubkey hash matches and signature is valid
  3. If both succeed, UTXO is spent

6. Bitcoin P2P Network

Overview

The Bitcoin P2P network is a decentralized mesh network where all nodes communicate directly without central servers. It enables distributed consensus and prevents single points of failure.

Network Architecture

Node1 ‚Üî Node2 ‚Üî Node3 ‚Üî Node4 ‚Üî Node5 (All connected to all others)

Peer Discovery

How Nodes Find Each Other

  • DNS Seeds: Well-known DNS servers return list of peer IPs
  • Hardcoded Seeds: Fallback IP addresses in client code
  • Address Relay: Peers share known addresses with each other
  • Bootstrap Nodes: Well-connected nodes that help new nodes

Network Properties

  • Decentralized: No central authority
  • Open: Anyone can join and run a node
  • Pseudonymous: IP addresses not required to match identity
  • Redundant: Thousands of copies of blockchain
  • Fault-Tolerant: Network survives node failures

UTXO Model

Unspent Transaction Output (UTXO)

Bitcoin uses UTXO model, not account-based. Each transaction creates outputs; to spend, you must completely consume and distribute outputs.

Transaction Example

Alice Sends 0.5 BTC to Bob

Input: Alice's previous UTXO of 1 BTC (from transaction X)

Output 1: 0.5 BTC to Bob's address

Output 2: 0.49 BTC back to Alice (change)

Fee: 0.01 BTC (input - outputs)

Transaction Propagation

  1. Alice creates and signs transaction
  2. Broadcasts to peers in P2P network
  3. Nodes validate transaction (signature, inputs exist, no double spend)
  4. Valid transactions added to mempool
  5. Miners select from mempool based on fees
  6. Transaction included in block
  7. Block added to blockchain

Transaction Validation Rules

  • Inputs must reference valid previous outputs
  • Signatures must be valid (ECDSA)
  • No double spending (same UTXO not spent twice)
  • Output amounts must not exceed input amounts (excluding fees)
  • Script must execute successfully
  • Transaction size within limits

8. Block Mining

What is Mining?

Mining is the process of validating transactions and creating new blocks in the Bitcoin blockchain. Miners compete to solve complex mathematical puzzles, and the winner gets to create the next block and receive rewards.

Mining Process

  1. Collect Transactions: Gather pending transactions from mempool
  2. Create Block Header: Include previous block hash, transactions root, timestamp, difficulty
  3. Find Nonce: Try different nonce values to find hash below difficulty target
  4. Broadcast Block: Send solved block to network
  5. Receive Reward: Get block reward (6.25 BTC) + transaction fees

Proof of Work (PoW)

Difficulty Target

Miners must find a block hash less than the target value. This requires trying many nonce values (quadrillions for Bitcoin).

Hash = SHA256(SHA256(block_header))
Hash < Target ‚Üí Valid block
Otherwise ‚Üí Try next nonce

Difficulty Adjustment

How Bitcoin Maintains 10-Minute Blocks

  • Every 2,016 blocks (~2 weeks): Difficulty recalculated
  • If blocks found faster: Difficulty increases
  • If blocks found slower: Difficulty decreases
  • Formula adjusts target to maintain ~10 minute average

Mining Pool

Why Pools?

Individual miners have negligible chance of finding block. Pools combine hash power and share rewards.

How Pools Work

  • Pool operator manages merged mining
  • Miners submit shares (partial solutions)
  • Pool pays based on shares contributed
  • Pool submits winning block to network

9. Block Propagation and Relay

Overview

Block propagation is the process of distributing newly mined blocks across the P2P network. Efficient propagation is critical for blockchain security and prevents unnecessary forks.

Block Propagation Flow

Miner finds block ‚Üí Broadcasts inv message ‚Üí Nodes request block ‚Üí Receives block data ‚Üí Validates ‚Üí Relays to peers

Propagation Timeline

Stage Time Description
Mining 0-10 min Miner finds solution
Announcement 0 sec Broadcasts inv (inventory) message
Download 0-2 sec Nodes request and receive block
Validation 0-1 sec Verify all transactions and signatures
Relay 1-5 sec Forward to connected peers
Network Coverage ~20 sec Reaches 90% of network

Network Propagation Challenges

Challenges

  • Network Latency: Messages take time to reach all nodes
  • Bandwidth Constraints: Block size vs network capacity
  • Validation Overhead: Full validation is computationally intensive
  • Orphan Blocks: Blocks received before parents
  • Network Partitions: Temporary disconnections cause forks

Optimizations for Fast Propagation

Compact Blocks (BIP 152)

  • Send only transaction IDs instead of full transactions
  • Receiver reconstructs from mempool
  • Reduces block size by 80-90%
  • Faster propagation across network

FIBRE (Fast Internet Bitcoin Relay Engine)

  • Relay network for low-latency block distribution
  • Uses erasure coding for efficient transmission
  • Private network separate from P2P network
  • Minimizes mining orphan rate

Block Relay Messages

Message Direction Purpose Size

block ‚Üí Send full block 1-4 MB
cmpctblock ‚Üí Send compact block 0.2-0.5 MB

Fork Resolution

What is a Fork?

When two blocks at same height are mined nearly simultaneously, nodes may differ on which block came first. The network automatically resolves by following the longest chain.

Resolution Process

  • Node receives competing block
  • Compares chain difficulty (cumulative PoW)
  • Reorg to longest valid chain if needed
  • Typically resolved in next block
  • Minority fork's transactions revert to mempool
