2. Blockchain as a Public Ledger

What is a Blockchain?

A blockchain is a distributed digital ledger that records transactions across multiple computers. It uses cryptographic hashing and consensus mechanisms to ensure immutability and prevent double-spending without a central authority.

How Blockchain Works as a Ledger

Transaction â†’ Validation â†’ Grouping â†’ Cryptographic Sealing â†’ Distribution â†’ Permanent Record

Key Properties

1. Decentralized

  • No central server or authority
  • Thousands of nodes maintain identical copies
  • Consensus determines valid transactions

2. Immutable

  • Cryptographic linking prevents modifications
  • Changing past block requires recalculating all following blocks
  • Prohibitively expensive to alter history

3. Transparent

  • All transactions publicly visible
  • Anyone can audit the entire history
  • Pseudonymous but traceable

4. Secure

  • Cryptographic signatures prevent tampering
  • Distributed nature protects against attacks
  • Consensus mechanism prevents unauthorized changes

The Longest Chain Rule

Definition

In case of a fork, the network always follows the chain with the most cumulative work (Proof of Work difficulty). This is called the "Longest Chain" or "Heaviest Chain".

Fork Resolution Process

  1. Two blocks found at same height (same parent)
  2. Network nodes receive blocks in different order
  3. Some follow chain A, others follow chain B
  4. Next miner extends longer chain
  5. Network converges to longest chain
  6. Shorter chain blocks become orphaned
  7. Transactions in orphaned blocks return to mempool

Longest Chain Properties

  • Consensus Mechanism: Longest chain = network consensus
  • Difficulty-Based: Based on cumulative PoW, not number of blocks
  • Security: Harder to attack longer chains (more work invested)
  • Finality: Blocks far back (~100 blocks) are effectively immutable
  • Natural Selection: Honest miners create longest chain fastest

6. Cryptographic Hash Functions

Definition

A cryptographic hash function is a mathematical algorithm that takes any input (of any size) and produces a fixed-size output (hash/digest). It's one-way: you cannot feasibly reverse it to get the original input.

Common Hash Algorithms

Algorithm Output Size Security Use in Blockchain
MD5 128 bits Broken Not used in crypto
SHA-1 160 bits Deprecated Legacy systems only
SHA-256 256 bits Secure Bitcoin, most blockchains
SHA-512 512 bits Very Secure Some cryptocurrencies
Keccak-256 256 bits Secure Ethereum
BLAKE2 256+ bits Very Secure Modern cryptocurrencies

Properties of Hash Functions

ğŸ” Deterministic

Same input always produces same output. Hash is predictable and repeatable.

âš¡ Fast Computation

Hash calculation is computationally efficient. Can hash gigabytes in seconds.

ğŸ” One-Way Function

Cannot reverse hash to get original input. Infeasible to find original from hash.

ğŸ“ Fixed Output Size

Output always same size regardless of input. Small change creates completely different hash.

ğŸ² Avalanche Effect

Tiny input change creates totally different output. Cannot predict changes.

â›” Collision Resistant

Virtually impossible to find two inputs with same hash. Requires 2^128 attempts (SHA-256).

Tamper Detection with Hash Pointers

Scenario: Attacker tries to modify Block 1

  1. Attacker changes transaction in Block 1
  2. Block 1's hash changes (avalanche effect)
  3. Block 2's previousHash no longer matches Block 1's hash
  4. Block 2 is invalid
  5. To fix Block 2, attacker must also change Block 2
  6. This invalidates Block 3, and so on...
  7. Attacker must recalculate entire chain (requires 51% attack)

8. Merkle Trees

Definition

A Merkle tree (hash tree) is a binary tree where each leaf node is a hash of a data block, and each internal node is a hash of its children's hashes. It provides efficient verification of large datasets.

Merkle Tree Structure

MERKLE TREE (4 Transactions)

                  Root Hash
                  (H1234)
                      /  \
            Internal Hash      Internal Hash
            (H12)             (H34)
              /  \             /  \
         Leaf Hash Leaf Hash  Leaf Hash Leaf Hash
         (H1)        (H2)            (H3)        (H4)
           |          |             |        |
        TX1       TX2            TX3      TX4

Construction Process

Step 1: Hash Transactions

Hash each transaction individually

  • H1 = SHA256(TX1)
  • H2 = SHA256(TX2)
  • H3 = SHA256(TX3)
  • H4 = SHA256(TX4)

Step 2: Hash Pairs

Hash each pair of hashes

  • H12 = SHA256(H1 + H2)
  • H34 = SHA256(H3 + H4)

Step 3: Hash Parents

Continue until single root hash

  • MerkleRoot = SHA256(H12 + H34)

Properties of Merkle Trees

ğŸ” Efficient Verification

Can verify dataset without downloading entire tree. Only need O(log n) hashes.

ğŸ¯ Targeted Verification

Can verify specific transaction without verifying all. Useful for SPV wallets.

ğŸ“š Compact Proof

Proof of inclusion requires only log(n) hashes, not all data.

â›” Tamper Detection

Any change in leaf changes all ancestor hashes. Tampering is immediately visible.

Merkle Proof Example

Verify TX2 in Block

Given: Root Hash (H1234) and TX2

Need to verify: TX2 is in block

Proof Required:

  • H1 (hash of TX1)
  • H34 (hash of subtree with TX3, TX4)

Verification:

  1. Calculate H2 = SHA256(TX2)
  2. Calculate H12 = SHA256(H1 + H2)
  3. Calculate MerkleRoot = SHA256(H12 + H34)
  4. Compare with known MerkleRoot - if match, TX2 is verified!

Merkle Trees in Bitcoin

  • Block Header: Contains MerkleRoot in 32 bytes
  • Transaction Verification: SPV nodes verify transactions without full blocks
  • Mobile Wallets: Use Merkle proofs for efficient verification
  • Scalability: Enables verification without storing all data

A permissioned blockchain (also called private or consortium blockchain) restricts who can participate in the network, create blocks, and validate transactions. It requires authentication and authorization.
